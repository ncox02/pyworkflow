A real session, step by step

Here's what a typical work session looks like — start to finish. Not theory. This is literally what happens.

1

Open your project in the IDE

You open a folder in VS Code or PyCharm. The IDE reads your project structure — your Python files, your configuration, your dependencies. It immediately starts providing: syntax highlighting, autocomplete, error detection.

The IDE knows your project structure before you've typed a single character.
2

Activate your virtual environment

Before writing any code, you activate your project's virtual environment. This tells Python "use the packages installed for this project." Your IDE may do this automatically if it detects a .venv folder. If not, you run source .venv/bin/activate in the terminal.

Your IDE's Python interpreter setting determines which environment is active. Get this right and everything else follows.
3

Write code — with AI alongside

You start writing. If you have Copilot, it suggests completions as you type — sometimes a line, sometimes a whole function. If you're stuck, you open Claude in a browser tab or the IDE's chat panel and ask a question. The AI can see your code if you paste it, or in some tools it can read your files directly.

AI is not a replacement for understanding — it's a very fast research assistant and rubber duck. The more context you give it, the more useful it becomes.
4

Run and test your code

You run your file (F5 in VS Code, Shift+F10 in PyCharm, or python myfile.py in the terminal). If something breaks, the error appears in the terminal. You read the traceback, ask the AI what it means if needed, fix it, and run again. This loop — write, run, fix — is the core of programming.

The built-in terminal in your IDE and your virtual environment need to agree — make sure you activated the right environment before running.
5

Commit your changes with Git

When your code is working and you want to save a snapshot, you make a Git commit. You write a short message describing what changed ("Added data cleaning function" or "Fixed bug in rate calculator"). This is now permanently recorded in your project's history — you can always return to this exact state.

git add .
git commit -m "Add data cleaning function"
Commit often. Small, well-labeled commits are infinitely more useful than one big commit at the end of the day.
6

Push to GitHub

You push your commits to GitHub. Now your code is backed up, visible to collaborators, and the history is safely off your machine. If you're working solo, this is just good hygiene. If you're working with others, this is where collaboration begins.

git push origin main
GitHub also stores your project's Issues (bugs, feature requests) and Pull Requests (proposed changes to review before merging).
7

Collaboration via Pull Requests

If working with others: they clone the repo, make their changes on a branch, push to GitHub, and open a Pull Request. You review their code on GitHub, leave comments, request changes, and eventually merge it. Git makes sure your changes and their changes don't overwrite each other.

Pull Requests are the primary way professional teams review and discuss code before it goes into the main branch. It's a quality gate, not just a formality.

Where AI fits in — specifically

AI doesn't replace any part of this workflow. It makes every part faster. Here's exactly where it helps.

While writing
Inline autocomplete (Copilot, JetBrains AI)

Suggests the next line, the next function, or generates a whole block from a comment. Works best for repetitive patterns and standard library usage.

When stuck
Chat-based AI (Claude, ChatGPT)

"Why is this throwing a KeyError?" or "How do I read a CSV file in pandas?" — paste your code, describe the problem, get an explanation and a fix. Far faster than searching Stack Overflow.

Before writing
Design and architecture (Claude, ChatGPT)

"How should I structure this project?" or "What's the best way to handle this kind of data?" AI is excellent at high-level design conversations — before any code is written.

After writing
Code review (Claude, Copilot chat)

"Review this function — is there a better way to write it?" or "Are there any bugs here?" AI can catch issues a human reviewer might miss and suggest cleaner approaches.

For large tasks
Agentic AI (Claude Code, Cursor Agent)

Give the AI a task ("write tests for all my functions" or "refactor this module to use a class"), and it reads your files, writes the code, and places it where it belongs — with you reviewing the result.

What a real project looks like

The structure of a Python project is not random. There are conventions that every developer recognizes — which makes it easy to understand any project the moment you open it.

my-project/
.venv/ ← virtual environment (never committed to Git)
src/
main.py ← entry point
analysis.py
utils.py
tests/
test_analysis.py
.gitignore ← tells Git what NOT to track
requirements.txt ← list of packages needed
README.md ← what this project is and how to run it
.gitignore — This file tells Git which files to ignore. Your .venv folder should always be here — it's huge, machine-specific, and reproducible from requirements.txt. So is .env (environment variables, often containing API keys — never commit these).
README.md — The homepage of your repository on GitHub. Write what the project does, how to install dependencies, and how to run it. This is the first thing anyone reads.
requirements.txt — The list of packages your project needs. Anyone cloning the project runs pip install -r requirements.txt to get everything installed. The equivalent with poetry is pyproject.toml.
