Classic Stack Setup

VS Code + venv + pip + GitHub Copilot + GitHub

1

Install VS Code

Download from code.visualstudio.com. Install and open it. You'll see a welcome screen.

VS Code is a folder-based editor. You'll always open a folder (your project), not individual files.
2

Install the Python extension

Click the Extensions icon in the left sidebar (looks like four squares). Search for "Python" — the one by Microsoft has millions of installs. Click Install. This gives VS Code Python understanding: syntax highlighting, autocomplete, environment detection, and a debugger.

3

Install Python

Download from python.org. On Windows, check "Add Python to PATH" during installation — this is important. On Mac, Python may already be installed, but download the latest version from python.org for a clean setup.

Verify the installation by opening your terminal and typing:

python --version

You should see a version number like Python 3.12.3.

4

Create your first project and virtual environment

Open VS Code. Go to File → Open Folder. Create a new folder called something like my-project and open it. Then open the built-in terminal (Terminal → New Terminal) and run:

# Create the virtual environment
python -m venv .venv

# Activate it (Mac/Linux)
source .venv/bin/activate

# Activate it (Windows)
.venv\Scripts\activate

You'll see (.venv) appear at the start of your terminal prompt — that means it's active.

VS Code may show a notification asking "Do you want to set up a Python environment?" — click Yes and select the .venv you just created. Now VS Code knows which Python to use.

5

Install packages with pip

With your virtual environment active, install packages and they'll be isolated to this project:

pip install pandas requests

# Save your dependencies to a file
pip freeze > requirements.txt
Always activate your virtual environment before installing packages. If you install without activating, packages go into your global Python — exactly what you're trying to avoid.
6

Add GitHub Copilot

In VS Code's Extensions panel, search for "GitHub Copilot" and install it. You'll be prompted to sign in with a GitHub account. Copilot requires a subscription ($10/month) — but is free for verified students via education.github.com.

Once installed, Copilot shows suggestions in gray as you type. Press Tab to accept a suggestion, or keep typing to ignore it.

7

Connect to GitHub

See the Universal: Git & GitHub section below — the setup is the same regardless of which IDE you're using.

Best for pure Python

Python-First Stack Setup

PyCharm + venv (auto) + JetBrains AI + GitHub

1

Download PyCharm

Go to jetbrains.com/pycharm. Download Community (free) or Professional. Students get Professional free at jetbrains.com/student — just sign up with a .edu email.

Community covers most Python development. Professional adds: web frameworks (Django, Flask), database tools, remote development, and JetBrains AI.

2

Create a new project

Open PyCharm → New Project. Choose a location. PyCharm will ask about the interpreter — select "Create new virtual environment." It creates and manages a .venv folder automatically. You never have to activate or deactivate it manually — PyCharm handles this for you.

This is one of PyCharm's biggest advantages over VS Code: virtual environment management is completely handled for you, with a UI to manage packages without touching the terminal.
3

Install packages (two ways)

Via terminal: PyCharm has a built-in terminal that automatically has the virtual environment active. Just type pip install pandas.

Via GUI: Go to Settings → Project → Python Interpreter → the + button. Search for any package and install it with a click. No terminal required.

4

Enable JetBrains AI (Pro only)

In PyCharm Pro, JetBrains AI is available in the right sidebar. It has access to your entire project — open files, project structure, error logs. Click the AI chat icon to open it. You can ask questions, have it generate code, explain functions, or review your changes.

Community edition doesn't include JetBrains AI, but you can always use Claude or ChatGPT in a browser tab alongside it.
5

Explore PyCharm's Python-specific features

Once you're set up, explore what makes PyCharm different:

  • Right-click any function → "Go to Definition" jumps to where it's defined, even inside libraries
  • Ctrl/Cmd+Click on any name does the same
  • The Debugger (click the bug icon) lets you pause code mid-execution and inspect any variable
  • Refactoring (Right-click → Refactor → Rename) safely renames a variable everywhere it's used
Cutting-edge

AI-First Stack Setup

Cursor + uv + Claude + GitHub

1

Install Cursor

Download from cursor.com. If you already use VS Code, Cursor will ask to import your settings, extensions, and keybindings. This makes the transition seamless — it feels identical to VS Code until you start using the AI features.

2

Install uv

uv replaces both pip and venv. Install it with a single command:

# Mac/Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# Windows (PowerShell)
powershell -c "irm https://astral.sh/uv/install.ps1 | iex"

Then create a project and environment:

uv init my-project
cd my-project
uv add pandas requests

uv creates a virtual environment automatically and manages it for you.

3

Set up Claude in Cursor

Cursor supports multiple AI models. In Cursor's settings, you can select Claude (Sonnet or Opus) as the underlying model for all AI features. You'll need an Anthropic API key — get one at console.anthropic.com.

Alternatively, Cursor's own subscription gives you access to multiple models including Claude without needing a separate key.

4

Learn Cursor's AI features

Cursor has several distinct AI interaction modes:

  • Inline completion — like Copilot, suggests code as you type
  • Cmd+K — edit mode: highlight code, press Cmd+K, describe the change, and Cursor rewrites it
  • Cmd+L — chat mode: ask questions about your code, with full file context
  • Agent mode — give a larger task, and Cursor reads files, writes code, and makes changes across multiple files autonomously
Agent mode is powerful but requires review. Always read what the agent wrote before accepting it — it can make mistakes, especially in larger codebases.

Git & GitHub Setup

The same process regardless of which IDE you use.

1

Install Git

Download from git-scm.com. On Mac, Git may already be installed — type git --version in your terminal to check. On Windows, the installer will also install Git Bash (a terminal that makes Git easier to use on Windows).

# Configure your identity (one-time setup)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
2

Create a GitHub account

Go to github.com and create a free account. Your username will be public — treat it like a professional identifier. Many developers use their GitHub profile as a portfolio.

3

Initialize Git in your project

In your project folder's terminal:

git init
git add .
git commit -m "Initial commit"
4

Create a .gitignore file

Before your first commit, create a file called .gitignore in your project root. This prevents Git from tracking files you don't want in your history — especially your virtual environment and any files with API keys or passwords.

# .gitignore
.venv/
__pycache__/
*.pyc
.env
.DS_Store
Never commit API keys, passwords, or secrets to Git. Even if you delete them in a later commit, they remain in the history. Use a .env file for secrets and add it to .gitignore.
5

Push to GitHub

On GitHub, click "New repository." Give it a name. Don't initialize with a README (since you already have code). Copy the commands GitHub shows you — they'll look like:

git remote add origin https://github.com/yourusername/your-repo.git
git branch -M main
git push -u origin main

Your code is now on GitHub. Every future commit can be pushed with just git push.

6

The daily Git rhythm

After setup, the loop you'll use every day is simple:

# See what changed
git status

# Stage your changes
git add .

# Commit with a message
git commit -m "Describe what changed"

# Push to GitHub
git push
VS Code and PyCharm both have built-in Git UIs — you can do all of this with clicks instead of commands. The Source Control panel in VS Code (left sidebar) shows your changes and lets you commit and push without the terminal.

Common pitfalls — and how to avoid them

These are the mistakes almost everyone makes at least once. Knowing them in advance will save you hours of confusion.

⚠

Installing packages without activating your virtual environment

If your terminal prompt doesn't show (.venv), your virtual environment isn't active. Installing packages now puts them in your global Python, which defeats the entire purpose. Always check before you pip install.

Fix: source .venv/bin/activate (Mac/Linux) or .venv\Scripts\activate (Windows)
⚠

Committing your .venv folder to Git

The virtual environment folder is large, machine-specific, and completely reproducible from your requirements.txt. Committing it bloats your repository and causes errors when others try to use it. It should always be in .gitignore.

Fix: Add .venv/ to your .gitignore before your first commit.
⚠

Committing API keys or passwords

Even a brief push of a file containing an API key is a security risk. GitHub scans for common patterns. If it happens, assume the key is compromised and rotate it immediately.

Fix: Store secrets in a .env file and add it to .gitignore. Use a library like python-dotenv to load it.
⚠

VS Code using the wrong Python interpreter

VS Code may have multiple Python installations to choose from — your global Python, your .venv, or even a conda environment. If you installed a package and Python can't find it, your IDE is probably pointing at the wrong interpreter.

Fix: Press Cmd/Ctrl+Shift+P → "Python: Select Interpreter" → choose the one inside your .venv folder.
⚠

Forgetting to update requirements.txt

When you install a new package, your requirements.txt doesn't update automatically. If a teammate clones your repo and runs pip install -r requirements.txt, they won't have the new package.

Fix: Run pip freeze > requirements.txt whenever you install a new package, and commit it.
⚠

Writing huge, vague Git commit messages

"Fixed stuff" or "changes" are useless commit messages. Six months later, you won't know what changed or why. Small, specific, frequent commits are dramatically more valuable than occasional big ones.

Fix: Commit often. Write messages like "Add error handling to data loader" — one clear sentence describing what changed.
